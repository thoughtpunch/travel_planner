# Design: Preference-Weighted Landed-Cost Optimization

## Context
Layers on `add-flight-orchestrator`. Reframes the objective from "cheapest
validated party airfare" to "cheapest validated landed cost to the true
destination, reordered by elicited traveler preferences."

## The core conceptual model

### Cost is the spine; preferences bend the ranking, never the cost
Landed cost is computed honestly and completely first (airfare + ground transfer
+ any lodging). Only then do preferences reorder the set. Preferences never
discount, inflate, or otherwise mutate a cost figure. This prevents the failure
where a *desired* feature (e.g. a stopover) gets both its real cost AND a
ranking boost in a way that double-counts in its favor or hides its true price.

### One bookended scale, not two
Replaces the earlier separate "direction (Likert)" + "importance (0–5)" design.
A single scale per axis:

```
HARD NO ── strongly avoid ── avoid ── neutral ── desire ── strongly desire ── HARD YES
```

This is more OkCupid-faithful (answer + weight fused into one control) and less
elicitation burden. Cost: you lose the ability to say "mild preference I care
enormously about," which for travel is an academic distinction.

### The decisive insight: HARD NO and HARD YES are different operations
The earlier design treated both scale-ends as operations on the **result set**,
which made HARD YES on a "desire" axis look incoherent (it would hide the cheap
direct option). That was a category error.

- **HARD NO** operates on the **result set** — subtractive, a filter.
- **HARD YES** operates on the **itinerary structure** — additive, a constructor:
  it injects the desired feature as a leg and searches for it.

Once separated, both ends are coherent and the asymmetry that plagued earlier
drafts dissolves. A HARD YES stopover does not filter out direct options; it
*builds a stopover itinerary* and prices it end to end.

### Soft vs. hard desire are genuinely different controls
- **strongly desire** (soft) = attractor over a mixed set: both stopover and
  direct itineraries appear; stopover ones rank higher. Never a requirement.
- **HARD YES** = constructor: the user has decided to do a stopover; build it.

This makes the top of the soft range and the hard end meaningfully distinct,
not redundant intensities.

## Decisions contested during design (recorded so they aren't re-litigated)

1. **Ground-transfer data: hardcoded, not scraped (v1).** Rail/ferry fares are
   stable; a maintained table with `last_reviewed` dates is accurate enough and
   avoids a second scraping rabbit hole. (User agreed.)

2. **Stopover lodging: user-owned assumption, not silent LLM estimate.** An LLM
   estimating hotel cost would inject a confident-but-wrong, date-insensitive
   number into the honest cost spine — poisoning the exact thing the spine is
   supposed to protect. The number must be user-owned, labeled, overridable; an
   LLM may only *pre-fill a labeled guess*. Generalized to: no silent
   model-generated cost components ever.

3. **Friction is not dollarized.** No fabricated misery-to-money exchange rate.
   Friction is elicited as preference and shown as visible attribute columns;
   the user's judgment makes the money-vs-friction call against an honest cost.

## Failure modes this change prevents
1. Ranking by fare and picking a "cheap" flight that's expensive to actually use.
2. Double-counting a desired feature (cost + boost) so it looks falsely cheap.
3. A "desire" hard setting silently deleting the cheapest direct option.
4. A silent LLM hotel estimate corrupting the honest cost spine.
5. Blending cost and friction into one opaque score the user can't interrogate.
