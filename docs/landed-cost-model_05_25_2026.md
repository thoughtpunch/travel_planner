# Spec Delta: landed-cost-model

## ADDED Requirements

### Requirement: Landed cost replaces airfare as the cost of an itinerary
The cost of an itinerary SHALL be its **landed cost** — the total cost for the
full party to reach the true destination (Venice) — not airfare alone. Landed
cost SHALL be the sum of: validated party airfare + ground transfer to the true
destination + any forced or intentional stopover lodging.

#### Scenario: Cheaper fare loses on landed cost
- **WHEN** itinerary X (fare $900/person to Rome) and itinerary Y (fare
  $1,200/person to Venice) are compared
- **AND** X requires a Rome→Venice transfer (per-person rail cost) for 6 people
- **THEN** the system ranks them by landed cost, not fare
- **AND** Y MAY rank above X despite the higher fare

### Requirement: Ground transfer model per gateway
Each candidate arrival gateway SHALL carry a transfer model to the true
destination: available modes (train / ferry / bus / drive), per-person cost,
duration, number of transfers, and a `forces_overnight` flag derived from
realistic arrival time vs. last viable onward departure.

#### Scenario: Gateway with viable same-day transfer
- **WHEN** a Northern Italy gateway (e.g. Milan, Verona, Bologna, Treviso) is
  evaluated
- **THEN** the transfer model provides a same-day onward option with cost and
  duration
- **AND** `forces_overnight = false`

#### Scenario: Gateway whose arrival forces an overnight
- **WHEN** an itinerary arrives too late for any viable same-day onward transfer
- **THEN** `forces_overnight = true`
- **AND** a lodging cost component is added to landed cost for that night

### Requirement: Ground-transfer data is a hardcoded table for v1
Ground-transfer cost/duration figures SHALL be a maintained hardcoded table per
gateway (v1), NOT scraped live. Each entry SHALL be labeled with a
`last_reviewed` date.

#### Scenario: Transfer figure shown
- **WHEN** a transfer cost contributes to landed cost
- **THEN** it is shown as a labeled assumption with its `last_reviewed` date

### Requirement: Cost components are honest, explicit, and user-overridable
Any estimated component entering landed cost SHALL be (a) user-overridable,
(b) visibly labeled as an estimate with its source, and (c) never silently
model-generated. An LLM MAY pre-fill a suggested value but it SHALL be labeled
"rough estimate, verify" and SHALL NOT enter the cost total as if measured.

#### Scenario: Stopover lodging assumption
- **WHEN** an itinerary includes a stopover requiring lodging
- **THEN** the lodging cost uses a user-owned per-night family-room assumption
- **AND** the value is displayed wherever landed cost appears, e.g.
  "stopover lodging assumption: $X/night (your estimate)"
- **AND** if an LLM pre-filled it, the label states it is an unverified estimate

#### Scenario: Forbidden silent estimate
- **WHEN** any cost component would be generated by a model without a label or
  without user override capability
- **THEN** this is a defect; the system SHALL NOT enter such a value into landed
  cost
