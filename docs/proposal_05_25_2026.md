# Change Proposal: Preference-Weighted Landed-Cost Optimization

## Status
Proposed

## Why

The base orchestrator (`add-flight-orchestrator`) ranks itineraries by validated
party **airfare**. That is the wrong objective. The user does not care about
fare; they care about the **total landed cost of arriving at their true
destination (Venice)**, where airfare is only one term. A $1,200 fare straight
to Venice can beat a $900 fare to Rome once the Rome option's train, transfers,
and a possible forced overnight for a party of 6 are added in.

Two distinct gaps must close:

1. **Landed cost, not airfare.** The objective function must include ground
   transfer to the true destination and any forced/intentional overnight
   lodging — priced as first-class components, honestly, before any ranking.

2. **Friction is per-traveler and cannot be guessed.** How much a long transfer,
   a long layover, a plane change, a red-eye, or a 24h+ stopover matters is a
   *preference*, and for stopovers it is genuinely bidirectional — the same
   feature is a disqualifier for one traveler and a desired rest-break for
   another. The system must **elicit** these preferences (OkCupid-style: what you
   want + how much it matters) rather than assume them, and use them to reorder
   an honestly-priced result set.

The design principle binding the two: **cost is the honest spine; preferences
bend the ranking around it but never alter the cost number.**

## What Changes

- NEW capability: `landed-cost-model` — extends the cost of an itinerary beyond
  airfare to include ground transfer to the true destination and stopover
  lodging, with explicit, user-owned, labeled assumptions.
- NEW capability: `preference-elicitation` — a single bookended preference scale
  per axis (HARD NO → soft avoid → neutral → soft desire → HARD YES), with
  global defaults and per-leg overrides, where the two ends are categorically
  different operations.
- NEW capability: `preference-scoring` — applies preferences to reorder the
  honestly-priced result set; specifies the no=filter / yes=construct duality
  and the rule that scoring never mutates cost.
- MODIFIED: `itinerary-orchestration` ranking requirement — rank by validated
  landed cost (not airfare), then apply preference scoring; support
  preference-driven stopover-leg construction.

## Impact

- Affected specs: NEW `landed-cost-model`, `preference-elicitation`,
  `preference-scoring`; MODIFIED `itinerary-orchestration`.
- Affected code: ranking pipeline, query/structure generation (stopover-leg
  construction), config schema (preferences, assumptions), UI (elicitation +
  result presentation showing cost spine + friction columns).
- Depends on: `add-flight-orchestrator` (this change layers on top of it).
- Cost data decision (confirmed with user): ground-transfer figures are a
  hardcoded per-gateway table for v1, NOT scraped. Stopover lodging is a
  user-owned assumption (LLM may pre-fill a labeled estimate).
- Out of scope: live hotel/rail pricing, booking, dollarizing friction into a
  fabricated misery-to-money rate.
